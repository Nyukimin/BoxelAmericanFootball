# 09_演出・アニメーション仕様書

- 目的: アメフトらしさを画面で伝える演出、アニメーション、可視化方針を定義する
- 関連: `02_ゲームコンセプト仕様書.md`, `04_試合進行・ルール仕様書.md`

## 1. 基本方針

演出は、ゲームを派手にするためだけのものではない。

本作では、演出は「なぜその結果になったか」を見せる教材機能でもある。

重要なのは、次の因果を画面で見せること。

```text
スナップ
  ↓
ラインがぶつかる
  ↓
走路ができる / 塞がる
  ↓
ボールが渡る / 投げられる
  ↓
守備が反応する
  ↓
タックル / キャッチ / 失敗 / ターンオーバー
  ↓
記録と解説
```

## 2. 表現上の制約

- 痛々しい表現をしない
- ぶつかってもポヨンと弾む
- 転んでもコロコロと可愛く転がる
- 怪我やケアは、白いテーピング帯など柔らかい表現にする
- 初心者が怖がらずに見られる表現にする
- プレイテンポを壊さない

## 3. 用語整理

「スクラム」はラグビーの用語であり、アメフトには存在しない。

本作では次の用語を使う。

| 用語 | 意味 |
|---|---|
| スクリメージライン | ボールを置く線 |
| スナップ | プレー開始の球出し |
| ラインの攻防 | OLと守備前列の押し合い、ブロック |

## 4. ボール視認性

どの視点でもボールを見失わないようにする。

必要な演出:

- ボールに常時光るマーカー
- ボール軌跡のトレイル
- ミニマップ上のボール位置
- パス・キック時の放物線
- ファンブル時の跳ね

## 5. 視覚的タックル

### 5-1. 目的

ランやパスの終わりに、守備が当たって止める動作を見せる。

数値上のタックル結果を、画面上で理解できるようにする。

### 5-2. 挙動

- プレー終了地点 `endZ` 付近の最寄り守備選手1体が、走者へ寄る
- 接触した瞬間、走者がポヨンと止まる
- 守備選手も少し反動を受ける
- 痛々しい転倒はしない

### 5-3. stats連動

- 実ヤードは `outcome()` で確定済み
- タックル位置・タイミングは結果に合わせる
- 走者の `speed` が高い場合、止まる位置が少し先に見える
- 守備が弱い場合、振り切りに近い見た目にする

### 5-4. 実装の勘所

- `updateAnim()` のラン/パス終端で処理する
- 最寄り `defense` を `carrier` 方向へ寄せる
- 接触時に小さなノックバックを入れる
- 既存の `finalize()` 呼び出しタイミングと整合させる

### 5-5. 制約

- 結果ヤードは変えない
- ファンブルと連動する場合のみ、結果に影響する

## 6. ラインの攻防

### 6-1. 目的

スナップ直後に味方ラインと守備前列が押し合う動作を見せ、采配と `power` が見た目で伝わるようにする。

### 6-2. 挙動

- スナップで両ラインが1〜2歩ぶつかる
- `power` が高い側が押し込む
- ランなら走路ができる、または塞がる様子を見せる
- パスならQBを守る壁として見せる

### 6-3. stats連動

- OLの `power` 平均と守備側の `power` を比較する
- 押し勝ち、押し負け、互角の見た目を変える

### 6-4. 実装の勘所

- `startPlay()` 冒頭に短いライン押し合い段を追加する
- `phase: "snap"` または前段アニメを追加する
- 既存のボール挙動の前に挟む
- アニメ時間が長くなりすぎないようにする

### 6-5. 制約

- 演出中心
- 結果は `outcome()` で確定する
- 初期版では個別マッチアップの詳細判定は行わない

## 7. ファンブル

### 7-1. 目的

タックル時にまれにボールを落とし、拾い合いで攻守が変わる新リスクを足す。

インターセプトに次ぐターンオーバー要素として、アメフトらしい劇的さを出す。

### 7-2. 発生条件

- ラン時
- キャッチ後のタックル時
- 低確率

### 7-3. 発生率

初期案:

- 発生確率: 3〜5%
- 相手リカバー率: 50%

調整:

- 走者の `power` が高いほどファンブルしにくい
- 強いタックルほど発生しやすい
- 多発させない
- クールダウン導入を検討する

### 7-4. 挙動

1. タックル時にボールがこぼれる
2. ボールが短く跳ねる
3. リカバー判定を行う
4. 相手が拾えばターンオーバー
5. 味方が拾えばその地点で継続

### 7-5. 物理演算の方針

結果は乱数で決める。物理エンジンで誰が拾ったかを判定しない。

理由:

- 物理任せはバランス調整が難しい
- 初心者に理不尽に見えやすい
- アメフトゲームでもリカバーは運要素として扱われる
- 単一HTML・依存なしの方針に合う

ボールの跳ねは、簡易擬似バウンドで表現する。

- バウンドごとに方向へ乱数
- 高さと速度を減衰
- 剛体物理エンジンは使わない

### 7-6. 実装の勘所

- `outcome()` に `fumble` フラグを追加する
- または `finalize()` 直前で判定する
- ターンオーバー時は既存の `endPossession()` を利用する

### 7-7. 制約

- 痛々しくしない
- 低頻度に抑える
- 新ルールなので実装は後段

## 8. キック演出

### 8-1. 目的

数値だけのFG/パントに、ボールを蹴って放物線で飛ぶアニメを足す。

### 8-2. 挙動

- キッカーがボールを蹴る
- ボールが放物線で飛ぶ
- FGではゴールポストへ向かう
- 成功/失敗で軌道を変える
- パントは高く遠くへ飛ぶ

### 8-3. ゴールポスト

- H型のゴールポストをフィールドに配置する
- `field()` にオブジェクトを追加する
- FG判定と見た目で同じ座標を使う

### 8-4. 実装の勘所

- 既存パスの放物線処理を流用する
- `kickPlay()` にアニメ段を追加する
- `ok` 成否に応じて軌道を分岐する
- 成否は既に確定済みの結果に合わせる

### 8-5. 制約

- 成否は `kick` 能力・距離・ミニゲーム結果で確定する
- 演出は結果に合わせる
- パントはミニゲーム化しない

## 9. FGミニゲームとの関係

FGミニゲームは `10_FGミニゲーム仕様書.md` に分離する。

本仕様では、FG成否後のボール軌道とキック動作を扱う。

FGミニゲームでは、ボール位置からゴールポストを見た角度とパワーで成否を決める。

## 10. アニメーションフェーズ案

```text
idle
  ↓
formation
  ↓
snap
  ↓
line_contact
  ↓
run / pass / kick
  ↓
defense_react
  ↓
tackle / catch / incomplete / fumble / score
  ↓
finalize
```

## 11. プレー解説リプレイとの接続

演出はリプレイ学習にも使う。

リプレイで見せるもの:

- スナップ
- ライン攻防
- 走路
- パス軌道
- 守備反応
- タックル
- ボールのこぼれ
- キック軌道

各選手に役割ラベルを表示する。

## 12. 実装順

推奨順:

1. キック演出
2. 視覚的タックル
3. ラインの攻防
4. ファンブル

教材価値を優先する場合は、視覚的タックルとラインの攻防を早める。

## 13. 今後の候補

| 要素 | 内容 | 本作への適性 |
|---|---|---|
| 2ポイントコンバージョン | TD後に2点を狙う | 高 |
| エクストラポイント | TD後のキック | 中 |
| キック/パントリターン | 蹴られた球を返す | 中 |
| QBスクランブル | QBが自分で走る | 中 |
| オーディブル | スナップ前に作戦変更 | 中 |
| プレイアクション | ランに見せてパス | 中 |
| 天候 | 雨/雪/風で結果が変わる | 高 |
| スタミナ/疲労 | 連続出場で能力低下 | 中 |
| スター選手の必殺 | 特定選手が能力UP | 高 |
| シーズン/育成 | 試合を重ねて成長 | 高だが規模大 |
| ペナルティ | 反則 | 教材上は重要、テンポ注意 |
| セーフティ | 自陣エンドゾーンで止められる | 低頻度 |
| 時計管理 | 2ミニッツ、クロック | 中 |
| ターンオーバーリターン | INT/ファンブル後に走る | 中 |

## 14. 非対象

- 物理エンジンの追加
- 痛々しい怪我表現
- NFLゲーム並みのリアル接触再現
- ピクセル単位の厳密な衝突判定
- 初期版での全11人完全AI挙動
