# オーケストレーター・プロンプト：残課題のTDD実装・E2E・自己評価ループ

使い方: 以下「===プロンプト===」以降を、Opus（オーケストレーター）への指示としてそのまま渡す。
Opus はサブエージェントを使って残課題を TDD で実装し、E2E で検証し、自己評価し、
ユーザー提供可能な水準に達するまで反復（切り返し）する。

===プロンプト===

## 0. あなたの役割

あなた（Opus）はこのリポジトリ「BoxelAmericanFootball」のオーケストレーターです。
下記の残課題を、TDDで実装し、E2Eで検証し、自己評価し、ユーザーに提供可能な水準に達するまで反復してください。
実装の細部はサブエージェント（Agent/Workflow）に委譲してよいが、品質ゲートの判断と統合はあなたの責任です。

## 1. 厳守する前提

- `CLAUDE.md` と `~/.claude/rules/` を遵守（最小差分・既存挙動を壊さない・絵文字禁止＝`[OK]`/`[NG]`/`[WARN]`・機密はコミットしない）。
- 動作確認済みの既存ロジック（stats反映／FG・パント／相手得点／試合終了）を壊さない。回帰したら即修正。
- 仕様の正は `docs/仕様追記_*.md`。実装は仕様に従う。乖離する場合は先に仕様を更新してから実装。
- コミットは日本語、`Co-Authored-By` を付与。push はユーザー承認後。
- 不確実な点は推測で埋めず「事実／仮説／未確認」を分けて報告。

## 2. 残課題（このループで仕上げる）

仕様書を参照して実装する。推奨順は各仕様書の「実装順の推奨」に従う。

1. キック演出（放物線＋ゴールポスト） … `docs/仕様追記_タックル_ライン_ファンブル_キック演出.md`
2. 視覚的タックル … 同上
3. ラインの攻防 … 同上
4. ファンブル（結果=乱数+power重み／跳ね=擬似バウンド／物理エンジン非採用） … 同上
5. FGミニゲーム（2段階タップ／ボール位置からの正しい透視／左右角度の幾何判定） … `docs/仕様追記_FGミニゲーム.md`

（FGミニゲームはFG周りなのでキック演出と束ねて扱ってよい。）

## 3. 開発方式：TDD（Red-Green-Refactor）

- まず純粋ロジックをテスト可能化する（タスク0）。`index.html` のインラインJSから、副作用のない計算
  （`outcome`、キック成否、相手ドライブ、ファンブル判定、FGの角度窓/成否、擬似バウンド更新 など）を
  テスト可能なモジュール（例 `gamelogic.js`）へ抽出し、`window.BoxelGame` 等で公開する。
  描画・DOM・THREE 依存は分離する（既存の `characters.js` 分離と同じ流儀）。
- 各機能で: 失敗するテストを先に書く(Red) → 最小実装(Green) → 整理(Refactor)。
- THREE 依存はモックする（`docs/調査/` の Node モック方式を踏襲）。

## 4. テスト戦略

### 単体テスト（必須）
- Node で純粋ロジックを検証。確率系は大数（10万回以上）で期待する方向・範囲を表明する。
- 例: speed↑でラン平均↑／catch↓で落球↑／FGは近い・kick高いほど成功↑かつ遠いと失敗あり／
  相手得点は陣地依存／ファンブルは低頻度かつ power で低減／FG角度窓はボール横位置で片寄る。
- `tests/` に置き、`node tests/*.mjs` で一括実行。全[OK]を必須。

### E2Eテスト（必須）
- Playwright 等で `index.html` をブラウザ起動し、UIフローを検証。
- ネットワーク制約を回避するため three.js をローカルにベンダリング（オフライン/同一オリジン）。
  本番の読み込み方法を壊さない形で（テスト時のみローカル参照に切替、または SRI付きCDN＋ローカルfallback）。
- 検証項目（DOM/状態/フロー中心。ピクセル厳密一致は求めない）:
  - スタート→隊形→プレー選択で HUD（ダウン/ヤード/スコア/シリーズ）が更新される
  - 4th で FG/パントが出る。FGミニゲームでメーターが動き、2タップで結果が出る
  - 攻守交代で相手スコアが動く／規定シリーズで試合終了画面→「もう一度」で再開
  - `console` の pageerror が 0
- 主要画面のスクリーンショットを保存（人手レビュー用）。WebGL描画の見た目の自動判定は必須にしない（人手確認に回す）。

## 5. 自己評価ルブリックと受け入れ基準（提供可能ライン）

各イテレーション末に自己採点（`[OK]`/`[改善中]`/`[NG]`）し、全項目[OK]で「提供可能」と判定する。

- 正確性: 仕様の受け入れ基準を満たす（機能ごとにチェックリスト化）
- 回帰なし: 既存の単体テストが全て緑
- 安定性: E2Eフローが完走、console error 0
- コード品質: `node --check` 通過、最小差分、命名/構造が周囲と整合、未使用コードなし
- 仕様整合: docs と実装が一致（乖離があれば docs 更新済み）
- UX: テンポを壊さない／痛々しくない（企画書 第7章）／初心者に分かる文言
- 既知の限界の明文化: WebGL見た目など人手確認事項を残す

未達なら、未達項目を次イテレーションのタスクに変換して継続。

## 6. オーケストレーション手順（1イテレーション）

1. 計画: 残課題から1機能を選ぶ（優先順）。受け入れ基準を箇条書き化。
2. Red: テスト作成サブエージェントに、失敗する単体テストを書かせる。
3. Green: 実装サブエージェントに最小実装させる。
4. Refactor: 整理（重複排除・命名・最小差分）。
5. E2E: E2Eサブエージェントにフロー検証させる。
6. 自己評価: 敵対的レビューサブエージェントに仕様逸脱・回帰・エッジを指摘させ、ルブリック採点。
7. 統合判定: あなたが結果を統合し「提供可能か」を判定。未達なら 1 に戻る（切り返し）。
8. 報告: イテレーション要約を提示し、ユーザーに確認/Push可否を問う。

- サブエージェントは `Agent`/`Workflow` で起動。独立作業は並列化。
- サブエージェントの最終出力は構造化（JSON）で受け取り統合する。

## 7. ループ終了条件（切り返しの止め時）

- 全残課題が「提供可能」基準を満たした → 完了報告。
- これ以上前進しない／同一失敗が続く（同一仮説は2回まで、3回目は視点を変える＝ルール準拠）→ ユーザーへエスカレーション。
- 1イテレーションごとに進捗（達成% = 100/50-99/1-49/0）を報告。

## 8. ガードレール

- 既存の動作確認済みロジックの挙動変更は、明示理由とテストの裏付けがある場合のみ。
- 外部ライブラリ追加は最小限・明示（Playwright 等テスト用は本番 `index.html` に持ち込まない）。
- three.js ベンダリングは本番の読み込みを壊さない形で。
- push はユーザー承認後。コミットは日本語、`Co-Authored-By` 付与。

## 9. 各イテレーションの報告フォーマット（必須）

```
目的: このイテレーションで何を仕上げるか
制約: 従うルール/前提
仮説: 実装/不具合の原因候補（優先度つき）
検証: 単体/E2Eの結果（[OK]/[NG]と数値）
判断: 提供可能か。未達項目は何か
次の一手: 次イテレーションのタスク or ユーザーへの確認
進捗: 達成% と残課題リスト
```
