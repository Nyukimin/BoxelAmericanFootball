# 17_グラフィック改善仕様書（現状診断と改善方針）

- 目的: ゲーム内容（ロジック・ルール・数値・結果）には一切触れず、グラフィック（描画品質・キャラ/ボールのモーション・演出）をレベルアップする
- 作成日: 2026年7月7日
- 診断方法: 本番サイトの実プレイ画面スクリーンショット（俯瞰/選手視点/FG）＋ index.html / characters.js の描画コード読解
- 制約:
  - gamelogic.js の関数・数値・乱数消費は不変（テスト 11/11 ファイル全緑を維持）
  - プレー結果値（ball.position.z 等）は演出前後で不変
  - アート方針（docs/02 §11: ぽっちゃり動物・ボクセル・パステル調・ポヨンと弾む・痛々しい表現なし）に準拠

## 1. 現状診断（何が悪いか）

### A. ライティング・シェーディング【最重要・費用対効果最大】

事実（index.html:2037-2048）:
- ライトは AmbientLight(0.75) + DirectionalLight(0.7) の2灯のみ
- 影が完全に無効（renderer.shadowMap 設定なし、castShadow/receiveShadow なし）
- マテリアルは MeshLambertMaterial（選手・ポスト）と MeshBasicMaterial（マーカー類）のみ
- renderer.outputEncoding 未設定（r128 デフォルトの linear のまま）

症状:
- 選手が地面に接地して見えず浮遊感がある（影がないため）
- ボクセルの各面がほぼ同じ明るさで、立体感・彫りの深さがない
- 画面全体が平板で「安いポリゴン」に見える

### B. キャラクターのモーション【体感品質を決める】

事実（characters.js makeAnimal / index.html アニメーション部）:
- 脚・腕は無名の box として Group に直接 add されており、参照が保持されていない（userData に保持されるのは head のみ）→ 手足を動かすことが構造的に不可能
- 走りは「本体ごと position.y を sin でホップ」のみ（index.html:1071 等）
- タックルは rotation.x の傾きのみ（index.html:2005 等）

症状:
- 選手が「走る」のではなく「滑りながら跳ねる」ように見える
- 全キャラが常に直立ポーズの置物で、生きて見えない
- タックル・キャッチ・ブロックに重み・タメがない

### C. ボールの可視化

事実:
- キャリア表示は「黄トーラスのリング＋橙コーン矢印」の空中マーカー（index.html:311-316）
- ボール本体（茶色 Sphere）はプレー中ほぼ存在感がない。スパイラル回転・軌跡なし

症状:
- 「誰がボールを持っているか」はマーカー頼みで、ボールを抱えて走る絵にならない
- パスの放物線が細く、ボールの飛翔が見どころになっていない

### D. フィールド・環境

事実・症状:
- 芝が単色グリーン＋縁の市松タイルのみ。実況感のある縞・ヤード番号・ハッシュマークがない
- エンドゾーンは青/オレンジのベタ1色（チーム名文字なし）
- 空が単色水色でスタンド・観客・地平線の景色が皆無 → スケール感と「試合」の空気がない
- ゴールポストは細い黄色 Cylinder の H 型のみ。ベース・パッドがなく、視点により歪んで見える

### E. カメラ・画面演出

事実・症状:
- カメラ切替（俯瞰/真上/選手視点）は補間なしの瞬間切替
- ヒット時のインパクト演出（軽いヒットストップ・画面シェイク・エフェクト）がない
- TD・FG成功時の祝祭演出（紙吹雪等）がない

## 2. 改善方針（どうしたいか）

### P1. ライティング＋影（A対応）
1. renderer.shadowMap を有効化し、DirectionalLight に castShadow、選手 castShadow / フィールド receiveShadow を設定（22体規模なら負荷は許容範囲。重い場合は足元の丸い半透明ブロブ影に代替）
2. AmbientLight を HemisphereLight（空色×芝色）に置き換え、面の向きで色味が変わるようにする
3. renderer.outputEncoding = sRGBEncoding を設定し、パステル調の発色を正しくする
4. 期待効果: 接地感・立体感が出て、全シーンの見た目が一段引き上がる

### P2. 手足モーション（B対応）
1. characters.js の makeAnimal で legL/legR/armL/armR を userData に参照保持（寸法・見た目は不変）
2. 走行中: 脚の前後スイング（sin 逆位相）＋腕の逆スイング＋軽い前傾。移動距離・座標は既存値のまま
3. 待機中: ゆっくりした呼吸（胴のごく小さい scale 揺れ）
4. タックル: 構え（タメ）→突進→ヒット時にスクワッシュ＆ストレッチ（ポヨン。アート方針適合）
5. TD: 両腕を上げてジャンプ＋その場回転
6. ポジション別体格差: 全ポジション共通の素体をやめ、role別の比率係数（横幅・身長・胴の厚み）で見た目を差別化する。OL/DL=幅広ずんぐり、TE/LB=やや大柄、RB=低くコンパクト、QB=標準、WR/CB/S=細身で手足長め、K/P=細身標準。座標・速度・判定など数値は不変（見た目のみ）。制約: 選手間隔2.0ydで重ならない素体幅の上限（characters.js のコメント参照）を超えないこと

### P3. ボールの絵作り（C対応）
1. キャリアはボールを脇に抱える（ボール mesh をキャリアの手元に attach。結果座標は不変）
2. パス中はスパイラル回転＋薄い軌跡（トレイル）を描画
3. キャリアマーカーは空中リングから「足元の柔らかいパルスリング（地面投影）」へ変更

### P4. フィールドの情報量（D対応）
1. 芝を5ヤードごとの明暗2トーン縞に
2. ヤード数字（10,20,...50）とハッシュマークを追加
3. エンドゾーンにチームカラー＋簡易文字
4. ゴールポストを太く・ベースパッド付きに（H型は維持）

### P5. スタジアム環境（D対応・後段）
1. サイドライン外に簡易ボクセルスタンド（色ブロックの観客表現）
2. 空にボクセル雲、遠景に低いスタンド壁で地平線を隠す
3. ベンチ・チームエリアの小物

### P6. カメラ・演出強化（E対応・後段)
1. カメラ切替を短いイージング補間に
2. ヒット時 0.1 秒前後のヒットストップ＋ごく軽い画面シェイク
3. TD/FG成功時に紙吹雪パーティクル＋ゴールポストのきらめき

## 3. 実施順序と検証

- 実施順: P1 → P2 → P3 → P4 → P5 → P6（各フェーズ完了ごとにスクリーンショット before/after 比較）
- 検証:
  - 各フェーズ後に node tests/run.mjs 全緑を確認（ロジック不変の担保）
  - claude-in-chrome によるブラウザ目視（俯瞰/真上/選手視点/FG の4視点）
  - FPS が体感低下しないこと（特に P1 の影と P5 のスタンド）
- 注意: Three.js は r128（2021年）。sRGBEncoding 等は r128 の API 名で実装すること（最新版の記事の API 名と異なる場合がある）

## 5. 残課題（次回）

- リアルタイム再生での最終目視（紙吹雪・ヒットストップ・走りモーションの滑らかさ・カメラ補間）。ブラウザタブが可視状態でないと requestAnimationFrame が止まるため、タブを前面にした状態で確認する
- 雲がフィールド上空に重なって見える件の位置調整（好みで場外上空へ）
- FGミニゲーム画面の背景（キックビュー）でのエンドゾーンスタンドの見え方の微調整
- ヘルプ画像は雰囲気イラストのため、正確な配置図・ルート矢印は絵に含まれない（キャプションの文章で補完済み）。将来、コード描画のSVG図解を足す余地あり
- push時、画像を含む大きめのパックはサンドボックス回線でSSL切断しやすい。`http.version=HTTP/1.1` 指定と、不要な中間バイナリ(削除予定の画像)を履歴に残さないコミット設計で回避した
