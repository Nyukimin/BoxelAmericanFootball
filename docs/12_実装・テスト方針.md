# 12_実装・テスト方針

- 目的: 実装構成、TDD、E2E、品質ゲート、公開方針を定義する
- 対象: 開発者、実装オーケストレーター、レビュー担当

## 1. 技術構成

### 1-1. プラットフォーム

- 1つのHTMLファイルで完結するブラウザゲームを基本とする
- インストール不要
- リンクを開くだけで遊べる
- PC・スマホ両対応

### 1-2. 描画

- Three.js による3Dボクセル表現
- 立方体の組み合わせでキャラクターとフィールドを作る

### 1-3. 物理表現

- ボールの放物線、バウンド、転がりを簡易物理・擬似物理で表現する
- 物理エンジンは追加しない
- ファンブルの拾得結果は乱数で決める

### 1-4. 方針

まず小さく動くものを作り、遊びながら育てていく。

## 2. ファイル構成

推奨構成:

```text
index.html
characters.js
gamelogic.js
tests/
  run.mjs
  *.test.mjs
  headless/
    flow.mjs
docs/
  00_INDEX.md
  ...
```

| ファイル | 役割 |
|---|---|
| `index.html` | UI、描画、イベント、統合 |
| `characters.js` | キャラ設定、見た目ビルダー |
| `gamelogic.js` | 副作用のない純粋ロジック |
| `tests/` | 単体テスト、ヘッドレスE2E |
| `docs/` | 仕様書 |

## 3. 実装の基本原則

- 最小差分
- 既存挙動を壊さない
- 教材目的を崩さない
- ゲーム性を削がない
- 痛々しくしない
- 用語を誤用しない
- 未確認事項は未確認と書く
- 仕様と実装が乖離する場合は、先に仕様を更新する

## 4. 純粋ロジック分離

`index.html` のインラインJSから、副作用のない計算を `gamelogic.js` へ抽出する。

対象例:

- `outcome()`
- キック成否
- 相手ドライブ
- 守備ドライブ
- ファンブル判定
- FG角度窓
- FG成否
- 擬似バウンド更新
- 反則判定
- 用語初出判定

公開例:

```js
window.BoxelGame = {
  outcome,
  kickPlay,
  defenseDrive,
  resolveFumble,
  resolveFgMiniGame,
  resolvePenalty
};
```

描画、DOM、THREE依存は分離する。

## 5. TDD方式

開発は Red-Green-Refactor で進める。

1. Red: 先に失敗するテストを書く
2. Green: 最小実装で通す
3. Refactor: 重複排除、命名整理、最小差分維持

確率系は大数で傾向を見る。

例:

- speedが高いほどラン平均が上がる
- catchが低いほど落球が増える
- kickが高いほどFG成功率が上がる
- 遠いFGほど失敗しやすい
- ファンブルは低頻度
- powerが高いほどファンブルしにくい
- FG角度窓は横位置で片寄る

## 6. 単体テスト

### 6-1. 実行方法

`node tests/run.mjs` で一括実行する。

`node tests/*.mjs` は最初の1ファイルしか実行されないため使わない。

### 6-2. 必須項目

- `outcome` のラン/パス結果
- stats反映
- FG成功率
- パント効果
- 相手得点
- 守備采配
- ファンブル判定
- FGミニゲーム判定
- 反則処理
- 試合終了

### 6-3. 確率テスト

確率系は10万回以上を目安に回し、期待する方向・範囲を確認する。

## 7. E2Eテスト

Playwright等で `index.html` をブラウザ起動してUIフローを検証する。

### 7-1. ネットワーク制約

- three.js はテスト時にローカルベンダリングする
- 本番の読み込み方法を壊さない
- CDN利用時はローカルfallbackを検討する

### 7-2. 必須フロー

- スタート
- 隊形選択
- プレー選択
- HUD更新
- 4thでFG/パント表示
- FGミニゲームの2タップ
- 攻守交代
- 相手スコア変化
- 規定シリーズ到達
- 試合終了画面
- 「もう一度」で再開

### 7-3. 品質条件

- `console` の `pageerror` が0
- 主要画面スクリーンショットを保存
- WebGL描画のピクセル厳密一致は求めない
- 人手レビュー用の見た目確認を残す

## 8. 品質ゲート

各イテレーション末に、以下を確認する。

| 項目 | 判定基準 |
|---|---|
| 正確性 | 仕様の受け入れ基準を満たす |
| 回帰なし | 既存単体テストが全て通る |
| 安定性 | E2Eフローが完走し、console error 0 |
| コード品質 | `node --check` 通過、未使用コードなし、最小差分 |
| 仕様整合 | docsと実装が一致する |
| UX | テンポを壊さず、初心者に分かる |
| 教材性 | マネージャー教材として意味がある |
| ゲーム性 | 選択と結果に手応えがある |
| 既知の限界 | 未確認事項を明示する |

判定表記は `[OK]`、`[WARN]`、`[NG]` を使う。

## 10. 残課題の推奨順

1. キック演出
2. 視覚的タックル
3. ラインの攻防
4. ファンブル
5. FGミニゲーム
6. 反則システム
7. 用語図鑑
8. プレー後の記録・分析UI強化

教材価値の即効性を優先する場合は、プレー後記録UIを早める。

## 11. オーケストレーション手順

### 11-1. 実行体制

- オーケストレーター: 計画、統合、品質判定、レビューを担当する
- コーダー: 実装・テスト作成を担当するサブエージェント
- サブエージェントを使う場合、実装細部は委譲しても、品質ゲートの判断と統合はオーケストレーターの責任とする
- 実行環境に `CLAUDE.md` や `~/.claude/rules/` がある場合は、それらを遵守する
- 出力表記は絵文字ではなく `[OK]`、`[WARN]`、`[NG]` を使う

### 11-2. 1イテレーション

流れ:

```text
目的設定
  ↓
受け入れ基準化
  ↓
Red: テスト作成
  ↓
Green: 最小実装
  ↓
Refactor: 整理
  ↓
E2E検証
  ↓
敵対的レビュー
  ↓
自己評価
  ↓
提供可能判定
  ↓
報告
```

報告フォーマット:

```text
目的:
制約:
仮説:
検証:
判断:
次の一手:
進捗:
```

## 12. コミット・push方針

- コミットメッセージは日本語
- `Co-Authored-By` を付与する
- 機密情報はコミットしない
- pushはユーザー承認後
- 不確実な点は、事実/仮説/未確認を分けて報告する

## 13. 公開方針

### 13-1. 公開

- Netlifyを使う
- HTMLファイルをドラッグ&ドロップして公開できる
- 無料枠を前提にする
- 企画書時点の想定では、無料枠は帯域100GB/月が目安。ただし上限つきクレジット制のため、大きな負荷時は注意する
- まずは遊べる試作を公開する

### 13-2. 収益化

広告は最後に検討する。

理由:

- 広告サービスの審査には独自ドメインが有利
- プライバシーポリシー、利用規約、お問い合わせ、運営者情報が必要
- ゲーム1ページだけでは審査に通りにくい
- 収益はアクセス数に依存する
- まずは遊びたくなる教材ゲームを作るべき

順番:

1. 動く試作を作る
2. 楽しい・可愛い・教材になるか確認する
3. Netlifyに公開する
4. 独自ドメインを取得する
5. 必要ページを用意する
6. 広告申請を検討する

## 14. 開発ロードマップ

| 段階 | テーマ | 作るもの |
|---|---|---|
| 第1段階 | 動いて可愛いを体験 | クオータービュー、フィールド、ボクセル選手、整列、スナップ、ボール物理、マーカー |
| 第2段階 | 采配の骨組み | 隊形→プレー選択UI、ダウン管理、前進判定、タッチダウン |
| 第3段階 | 視点と学習 | 選手視点、マスコットコーチ、段階的チュートリアル |
| 第4段階 | 可愛さと手触り | アニメーション、効果音、BGM、動物キャラ、演出 |
| 第5段階 | 教材強化 | 記録UI、用語図鑑、反則、マネージャーメモ |
| 第6段階 | ゲーム性強化 | 守備テル、タックル、ライン攻防、ファンブル、FGミニゲーム |

## 15. nodeチェック

実装時は最低限、次を通す。

```bash
node --check gamelogic.js
node tests/run.mjs
```

E2E導入後:

```bash
node tests/headless/flow.mjs
```

## 16. 既知の限界

- WebGLの見た目は自動判定しにくい
- ブラウザ目視確認が必要
- スマホ実機での操作感は別途確認が必要
- 物理挙動は擬似表現であり、厳密な再現ではない
- 公式ルールの細部は省略する

## 17. 非対象

- 外部物理エンジン導入
- サーバー必須構成
- 本格的なオンライン対戦
- 初期版での広告最適化
- 上級者向け複雑戦術シミュレータ
