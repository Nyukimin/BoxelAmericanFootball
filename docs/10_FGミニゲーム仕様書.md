# 10_FGミニゲーム仕様書

- 目的: フィールドゴールを、左右角度とパワーを狙う2段階タップミニゲームとして定義する
- 関連: `04_試合進行・ルール仕様書.md`, `09_演出・アニメーション仕様書.md`, `11_UI・画面仕様書.md`
- 状態: 設計済み・未実装

## 1. 目的

確率任せだったフィールドゴールを、プレイヤーが左右の角度と飛距離を自分で狙うミニゲームにする。

ボールを置いた位置から正しい見え方でゴールポストを見せ、横からのキックほど難しくなる駆け引きを出す。

## 2. 操作方式

採用方式は2段階タップ。

### 2-1. ステップ1: 左右の角度

角度マーカーが左右に往復する。

プレイヤーがタップまたはクリックすると、狙う向き `aim` が決まる。

中央に近いほどまっすぐ。

表示例:

```text
左 [-----|----▲----|-----] 右
```

### 2-2. ステップ2: パワー

ゲージが伸びる。

プレイヤーがタップまたはクリックすると、飛距離 `power` が決まる。

表示例:

```text
弱 [#######...] 強
```

### 2-3. 決定後

- キッカーがボールを蹴る
- ボールが狙った向き・強さで放物線で飛ぶ
- ポールの間を通れば成功
- 成功時は3点
- 成否にかかわらず攻守交代

## 3. カメラ・ポールの見え方

### 3-1. 重要方針

FG中のカメラは、正面固定にしない。

ボールの位置からゴールポストを見る。

### 3-2. カメラ配置

- ボールの少し後方
- 低め
- ゴール中心を見る

実装イメージ:

```js
camera.position = ballPos + (0, 約6, 約-14)
camera.lookAt(goalCenter)
```

### 3-3. 横位置の反映

ボールが横、つまりハッシュ寄りにあると、ゴールは画面の左右どちらかに寄り、斜めに見える。

横にあるほど、通せる左右角度の窓が狭く、片寄って見える。

この見え方がFGの難しさになる。

## 4. ゴールポスト配置

見た目と判定で同じ座標を使う。

| 要素 | 座標・値 |
|---|---|
| ゴール平面 | `z = 120` |
| 左アップライト | `x = -3` |
| 右アップライト | `x = 3` |
| クロスバー高さ | `y ≈ 3` |
| 形状 | H型 |

`field()` にゴールポストオブジェクトを追加する。

## 5. 成否判定

### 5-1. 入力値

| 値 | 意味 |
|---|---|
| `xb` | ボールの横位置。直近プレーの位置。なければ0 |
| `zb` | ボールの縦位置。現在の `los` |
| `θaim` | プレイヤーが止めた角度 |
| `power` | プレイヤーが止めたパワー |
| `kick` | キッカー能力 |

`xb` はハッシュ相当に `[-6, 6]` でクランプする。

### 5-2. 左右角度窓

各ポールへの方位角:

```js
θL = atan2(-3 - xb, 120 - zb)
θR = atan2( 3 - xb, 120 - zb)
```

プレイヤーの角度が次を満たせば左右成功圏。

```js
θL <= θaim <= θR
```

横にあるほど、この範囲は片寄り、狭くなる。

### 5-3. パワー判定

必要飛距離は、ボールからゴールまでの距離を基準にする。

```text
必要飛距離 = 120 - zb + α
```

`power` が必要飛距離以上なら飛距離OK。

### 5-4. 成功条件

```text
成功 = 左右OK かつ 飛距離OK
```

### 5-5. 失敗理由

| 失敗理由 | メッセージ例 |
|---|---|
| 左に外れた | 「左にそれた！」 |
| 右に外れた | 「右にそれた！」 |
| 飛距離不足 | 「届かなかった…！」 |

ボールは、判定結果と一致する軌道で飛ばす。

## 6. 能力値の反映

`kick` 能力は、難度に影響する。

| 能力反映 | 内容 |
|---|---|
| メーター速度 | `kick` が高いほど角度メーターを遅くする |
| 成功窓許容 | `kick` が高いほど左右判定に少し許容を足す |
| パワー安定 | 後段で、`kick` が高いほどパワー調整しやすくする |

初期案:

```js
meterSpeed = base * (2 - kick / 6)
tolerance = 0.01 * (kick / 6)
```

## 7. 天候の反映

天候は任意・後段。

横風を入れる場合は、`θaim` にオフセットを加える。

```js
θactual = θaim + windOffset
```

これにより、風で左右に流れる表現ができる。

## 8. 演出

### 8-1. 成功

- キッカーが蹴る
- ボールが放物線で飛ぶ
- アップライトの間を通る
- 3点加算
- 成功演出

### 8-2. 失敗

- 左右にそれる
- または手前で落ちる
- 失敗理由を表示
- 攻守交代

痛々しい表現はない。

## 9. フロー

```text
4thダウンでFGを選ぶ
  ↓
kickビューへ切替
  ↓
角度メーター表示
  ↓
1回目タップ
  ↓
パワーメーター表示
  ↓
2回目タップ
  ↓
成否判定
  ↓
キック軌道演出
  ↓
成功なら3点
  ↓
攻守交代
```

## 10. 調整パラメータ

| 項目 | 初期値・式 | 備考 |
|---|---|---|
| ゴール幅 | `x = ±3` | 通すべき左右幅 |
| ゴール奥行 | `z = 120` | 判定平面 |
| クロスバー高さ | `y ≈ 3` | 見た目用 |
| 角度メーター可動域 | `θcenter ± 0.28 rad` | 成功窓より広め |
| メーター速度 | `base * (2 - kick/6)` | kick高いほど遅い |
| 成功窓許容 | `+0.01 * (kick/6)` | わずかに広げる |
| ハッシュ範囲 | `xb in [-6, 6]` | 横位置 |

## 11. UI要件

- PCクリックとスマホタップの両対応
- 1回目と2回目の状態を明確に表示する
- 「角度」「パワー」を初心者にも分かる言葉で表示する
- 結果メッセージは短くする
- ミニゲーム中は他のカードUIを隠す
- キック後は通常UIに戻す

## 12. 純粋ロジック化

判定はテスト可能な純粋関数にする。

例:

```js
fgWindow({ xb, zb }) -> { thetaLeft, thetaRight, thetaCenter }
fgPowerRequired({ xb, zb }) -> number
resolveFgMiniGame({ xb, zb, thetaAim, power, kick }) -> {
  ok,
  reason,
  thetaActual,
  requiredPower
}
```

## 13. テスト観点

- 中央からは成功角度窓が左右対称になる
- 左ハッシュからは窓が片寄る
- 右ハッシュからも同様に片寄る
- 遠い位置ほど必要パワーが増える
- `kick` が高いほど成功しやすい
- 左右外れと飛距離不足の理由が分かれる
- キック後に攻守交代する

## 14. 制約・非対象

- パントはミニゲーム化しない
- 正面固定カメラにしない
- 実際の風や物理計算を厳密に再現しない
- 複雑なタイミング操作にしない
- 操作は2段階タップに限定する
